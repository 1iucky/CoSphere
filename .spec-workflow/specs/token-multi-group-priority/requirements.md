# 需求规格文档: 令牌多分组优先级管理

**项目**: New API - AI网关与资产管理系统
**功能**: 令牌多分组优先级管理
**创建日期**: 2025-11-30
**文档状态**: 草稿

---

## 1. 概述

### 1.1 功能背景

当前令牌管理系统仅支持单一分组配置,无法满足以下场景需求:
- 用户希望为令牌配置多个备用分组,提高服务可用性
- 用户需要控制不同分组的使用优先级,优先使用成本更低或性能更好的分组
- 系统在主分组不可用时,需要自动降级到备用分组

### 1.2 业务价值

1. **提高服务可用性**: 多分组配置提供自动故障转移能力
2. **优化成本控制**: 通过优先级控制,优先使用低成本分组
3. **改善用户体验**: 减少因单点故障导致的服务中断
4. **灵活性提升**: 用户可根据实际需求灵活配置分组策略

### 1.3 目标用户

- **终端用户**: 使用令牌调用AI模型服务的开发者
- **系统管理员**: 配置和管理令牌分组的运维人员

## 2. 功能需求

### 2.1 用户故事

#### US-1: 创建多分组令牌
**作为** 系统用户
**我想要** 在创建令牌时选择多个分组并设置优先级
**以便** 实现服务的高可用和成本优化

**验收标准**:
- GIVEN 用户在创建令牌页面
- WHEN 用户选择分组时
- THEN 系统应支持多选分组
- AND 用户可以通过拖动调整分组的优先级顺序
- AND 系统实时显示当前的优先级顺序
- AND 分组选择器附近显示功能说明提示文本
- AND 自动智能分组开关附近显示功能说明提示文本
- AND 保存后能够正确存储多分组配置

#### US-2: 编辑令牌分组优先级
**作为** 系统用户
**我想要** 修改现有令牌的分组配置和优先级
**以便** 根据实际运行情况调整分组策略

**验收标准**:
- GIVEN 用户在编辑令牌页面
- WHEN 用户修改分组配置时
- THEN 系统显示当前已配置的分组及其优先级
- AND 用户可以添加、删除分组
- AND 用户可以调整分组优先级顺序
- AND 分组选择器附近显示功能说明提示文本
- AND 自动智能分组开关附近显示功能说明提示文本
- AND 修改后能够正确更新数据库

#### US-3: 查看令牌分组配置
**作为** 系统用户
**我想要** 在令牌列表中清晰看到每个令牌的分组配置
**以便** 快速了解令牌的分组策略

**验收标准**:
- GIVEN 用户在令牌列表页面
- WHEN 用户浏览令牌列表时
- THEN 每个令牌显示其配置的所有分组
- AND 分组按优先级顺序展示 (如: group1 > group2 > group3)
- AND 优先级顺序清晰可读

#### US-4: 多分组优先级转发
**作为** 系统
**我想要** 按照用户设置的分组优先级顺序进行请求转发
**以便** 实现智能的负载分发和故障转移

**验收标准**:
- GIVEN 令牌配置了多个分组 [group1(priority=1), group2(priority=2), group3(priority=3)]
- WHEN 用户使用该令牌发起请求
- THEN 系统首先尝试使用 group1 的渠道
- AND 如果 group1 中不存在请求的模型,自动尝试 group2
- AND 如果 group1 请求失败/异常,自动尝试 group2
- AND 依此类推,直到成功或所有分组都失败
- AND 每次降级都应记录日志

#### US-5: 自动智能分组
**作为** 系统用户
**我想要** 启用自动智能分组功能
**以便** 在配置的分组都失败时,系统自动从其他可用分组中选择

**验收标准**:
- GIVEN 令牌配置了自动智能分组为"开启"
- WHEN 所有配置的分组都无法提供服务时
- THEN 系统自动从其他可用分组中按费率从低到高的顺序尝试
- AND 成功响应请求后记录实际使用的分组
- AND 如果自动智能分组为"关闭",则直接返回错误

## 3. 非功能需求

### 3.1 性能要求

- **响应时间**: 分组选择逻辑不应增加超过 50ms 的额外延迟
- **并发处理**: 支持至少 1000 QPS 的令牌验证和分组选择
- **缓存机制**:
  - 利用现有 Redis 缓存机制存储令牌信息(包括 GroupPriorities 字段)
  - 令牌验证优先从 Redis 读取,缓存未命中才查询数据库
  - 令牌更新/删除操作同步更新 Redis 缓存
  - 确保缓存与数据库数据一致性
- **数据库查询**: 在 Redis 启用的情况下,单次令牌验证通常只需 0 次数据库查询(缓存命中),缓存未命中时不超过 1 次查询

### 3.2 可用性要求

- **向后兼容**: 现有单分组令牌应能正常工作
- **数据迁移**: 提供平滑的数据迁移方案,不影响现有令牌
- **降级策略**: 多分组逻辑失败时,应降级到单分组模式

### 3.3 可维护性要求

- **代码清晰**: 新增逻辑应遵循现有代码风格和架构
- **日志完整**: 记录每次分组选择和降级过程
- **配置灵活**: 自动智能分组可在令牌级别配置

### 3.4 安全性要求

- **权限控制**: 用户只能配置自己有权访问的分组
- **参数验证**: 前后端都需验证分组配置的合法性
- **注入防护**: 防止 SQL 注入和 JSON 注入攻击

## 4. 约束条件

### 4.1 技术约束

- **后端**: 基于现有 Go + Gin 框架,不引入新的依赖库
- **前端**: 基于现有 React + Semi Design,尽量使用现有组件
- **数据库**: 兼容 PostgreSQL, MySQL 和 SQLite
- **存储格式**: GroupPriorities 字段使用 JSON 字符串存储

### 4.2 业务约束

- **最大分组数**: 单个令牌最多配置 10 个分组
- **优先级范围**: 优先级从 1 开始,数字越小优先级越高
- **数据一致性**: Group 字段保留,用于向后兼容和单分组场景

### 4.3 时间约束

- **开发周期**: 预计 2-3 个开发迭代
- **测试周期**: 充分的单元测试和集成测试
- **发布策略**: 灰度发布,逐步推广

## 8. 未来扩展

### 8.1 可能的增强功能

- **智能分组推荐**: 基于历史数据推荐最优分组配置
- **分组性能监控**: 展示每个分组的成功率、延迟等指标
- **分组配额管理**: 为每个分组设置独立的配额限制
- **分组时段策略**: 根据时间段自动切换分组优先级

### 8.2 暂不实现的功能

- **分组权重配置**: 按权重随机选择分组 (暂不需要)
- **分组条件路由**: 根据请求内容动态选择分组 (复杂度高)
- **跨令牌分组共享**: 多个令牌共享同一套分组配置 (需求不明确)

## 附录

### A. 术语表

- **分组 (Group)**: 渠道的逻辑分组,如 "default", "gpt-4-group" 等
- **优先级 (Priority)**: 分组的使用顺序,数字越小优先级越高
- **降级 (Fallback)**: 当前分组不可用时,自动切换到备用分组
- **费率 (Ratio)**: 分组的计费倍率,影响使用成本
